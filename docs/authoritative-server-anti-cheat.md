# 服务端权威与防作弊

## 1. 核心原理

服务端绝对权威 = 客户端不可信 = 所有核心逻辑都必须在服务端计算。

客户端只负责：

- 展示 UI
- 播放动画
- 收集用户输入

客户端不负责：

- 发牌
- 扣筹码
- 计算牌型
- 判定胜负
- 结算输赢
- 判定是否轮到自己行动

原因很简单：

- 客户端内存可以改
- 客户端包可以抓
- 客户端请求可以伪造
- 客户端构建可以被篡改

所以任何由客户端直接提交的 `chips / cards / result / winner` 都不能信。

## 2. 这个项目里必须服务端算什么

在 `backend-cpp/src/main.cpp` 里，真实状态都应该只存在服务端：

- 真实房间状态：`Room`
- 真实玩家状态：`PlayerState`
- 真实座位归属：`Seat + Session + GameplaySessionData`
- 真实牌堆：`room.deck`
- 真实公共牌：`room.community_cards`
- 真实手牌：`p->hand`
- 真实结算：`finish_hand()` + 手牌比较逻辑

客户端只拿到：

- 公共房间状态：`room_state`
- 公共对局状态：`game_state`
- 自己座位的私牌：`private_hand`

也就是说，客户端看到的是服务端计算结果的投影，不是事实来源。

## 3. 为什么客户端不能改筹码

客户端本地的筹码只是展示副本。

真实筹码只在服务端维护：

- 下注扣筹码：`place_bet()`
- 当前下注额：`current_bet`
- 当前最大下注：`room.current_max_bet`
- 最终胜负结算：`finish_hand()`

所以客户端就算发：

- “我要加 1000 筹码”
- “我的余额现在是 999999”

服务端也不会采用。

服务端只接受“动作意图”：

- `fold`
- `check`
- `call`
- `raise`
- `allin`

然后服务端自己决定这次动作是否合法、真实会扣多少筹码、结果如何广播。

## 4. 当前项目的权威校验流程

现在服务端动作处理已经按这个顺序走：

1. 客户端发送动作请求
2. 服务端确认当前 socket 绑定了已开始的房间
3. 服务端确认 seat 存在且属于当前登录用户
4. 服务端确认当前 seat 仍然由这个 socket 控制
5. 服务端确认轮到这个 seat 行动
6. 服务端确认动作类型合法
7. 服务端确认数值合法，比如不能在筹码不足时强行 raise
8. 服务端修改 `Room / PlayerState`
9. 服务端广播结果
10. 客户端只渲染服务端确认后的结果

## 5. 这次补上的反作弊硬校验

为了让“服务端权威”不只是概念，这次已经把这些运行时拦截补上：

- 非法动作不再静默忽略，而是明确返回 `Action rejected: ...`
- 未进入已开局房间时，不能发送行动请求
- 非当前座位拥有者，不能替别人的 seat 行动
- 非当前回合玩家，不能抢行动
- 面对下注时不能 `check`
- 筹码不足时不能伪造 `raise`
- 没筹码时不能伪造 `allin`
- 非玩家 seat 不能发送玩家动作

这意味着客户端就算通过抓包把请求改成异常值，服务端也只会拒绝，不会污染真实房间状态。

## 6. 发牌与结算为什么必须只在服务端

这个项目里真正的牌局计算只在服务端：

- 洗牌：`fresh_deck()`
- 发手牌：`deal_hole_cards()`
- 发公共牌：`deal_community()`
- 牌型比较：`evaluate7()`
- 胜者判定和分池：`finish_hand()`

客户端没有资格决定：

- 自己摸到了什么牌
- 这把谁赢
- 赢多少
- 自己的下注是否真的成功

客户端最多只能说：

- “我想 call”
- “我想 raise 200”

至于是否允许、是否筹码足够、是否真的进入底池，都由服务端决定。

## 7. 面试版答案

可以直接这样回答：

1. 服务端权威：客户端完全不可信，所有核心逻辑都在服务端计算，客户端只做展示和输入。
2. 防作弊：客户端不能决定牌、筹码、胜负、结算，所有这些都由服务端维护和广播。
3. 校验机制：每次动作都要校验身份、房间、座位归属、轮次、动作合法性和筹码数值，非法请求直接拒绝。

## 8. 这个项目后续还可以继续加强的点

- 给动作请求补 `turn_nonce`，防止旧包重放
- 给关键安全拒绝加审计日志，便于追踪异常客户端
- 给高频非法请求加限流或封禁策略
- 把更多安全规则单独抽成 `ActionValidator`
